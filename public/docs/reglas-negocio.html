Reglas de Negocio

Comportamiento esperado del sistema: IVA, matrículas, pagos, bonificaciones y validaciones. Si una regla cambia, actualizar aquí e impactar UI / API / BD.

IVA

  • El IVA se calcula después de aplicar todos los descuentos sobre la base imponible neta.
  • Solo aplica a ítems gravados; ítems exentos tienen IVA = 0.
  • La base imponible no puede ser negativa; mínimo es 0.
  • IVA y base imponible se redondean a 2 decimales por transacción.
  • Cambios en descuentos o bonificaciones recalculan el IVA automáticamente.

Estados de matrícula

EstadoCondición
pendientePagos acumulados < monto_min_reserva
parcialPagos acumulados ≥ monto_min_reserva y < monto_matricula
pagadoPagos acumulados ≥ monto_matricula
anuladoReverso total o anulación manual
  • La reserva solo es válida si el pago inicial ≥ monto_min_reserva.

Pagos

  • Se permiten pagos parciales; cada pago debe ser > 0 y ≤ saldo pendiente.
  • Pagos por transferencia no se imputan hasta confirmación.
  • No se permiten pagos sobre matrículas en estado anulado.
  • Todo reverso debe referenciar el pago original, con motivo y usuario responsable.
  • Saldo pendiente = total_matricula − pagos_confirmados − descuentos/bonificaciones.
Reverso de pagos

Un reverso no elimina el pago original; crea un movimiento de tipo egreso que referencia al pago original. Nunca se borra físicamente un registro de pago.

Bonificaciones

  • Bonificaciones y descuentos no son acumulables entre sí; se aplica el mayor.
  • Una bonificación solo aplica a matrículas en estado pendiente o parcial.
  • Requiere vigencia definida (fecha_inicio y fecha_fin); fuera del rango no aplica.
  • Pasa a estado no_pagada si vence el plazo sin que se complete el pago.
  • Se anula si la matrícula o el curso/promoción se anulan.
  • Un job automático corre diariamente para anular bonificaciones vencidas.

Interesados y Bitácora

  • No se duplican interesados por documento o email; se actualiza el existente.
  • Cada visita en bitácora requiere recepcionista asignado, o motivo si es NULL.
  • convertido_a_prematricula = 1 solo si existe matrícula asociada en estado pendiente o parcial.
  • fecha_fin_visita debe ser ≥ fecha_visita cuando se registra.
  • La bitácora se elimina de forma lógica (deleted_at), nunca física.

Usuarios y roles

  • El email es obligatorio y único por usuario.
  • El username es opcional; si existe, debe ser único.
  • Usuarios inactivos o bloqueados no pueden iniciar sesión.
  • Un usuario sin rol no tiene permisos de módulo hasta que se le asigne uno.
  • Cambiar el rol de un usuario invalida sus sesiones activas.

Centros educativos

  • El RUC es obligatorio y único por centro.
  • El nombre es obligatorio; el alias es opcional.
  • No se puede desactivar el último centro activo.
  • La eliminación es lógica; no se puede eliminar un centro con cursos, estudiantes o convocatorias activas.
  • Solo centros activos pueden recibir nuevos cursos o convocatorias.

Catálogo académico

  • Solo cursos activos (sin deleted_at) aparecen en el catálogo de inscripción.
  • Un curso debe tener al menos un nivel y una especialidad asignados.
  • No se puede eliminar un curso con promociones o matrículas activas.

Empresas

  • Si se registra RUC, debe ser único.
  • Se requiere al menos un contacto: email o teléfono.
  • Solo empresas activas pueden asociarse como empresa_origen de un interesado.
  • Inactivar una empresa no borra el historial de interesados o experiencias asociadas.
