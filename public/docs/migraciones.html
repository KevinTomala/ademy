Guía de Migración de Datos

Estructura, patrones y buenas prácticas para los scripts de importación masiva de promociones al sistema Ademy.

Aviso de privacidad

Los ejemplos de esta guía usan datos ficticios. Los scripts reales contienen datos sensibles de clientes y no deben ser compartidos públicamente. Acceder a los scripts de producción requiere permisos de administrador del sistema.

¿Qué es un script de migración?

Los scripts de migración (import_XXXX_NX.sql) permiten importar masivamente una promoción completa a la base de datos, incluyendo interesados, estudiantes, matrículas, pagos, entregas y formaciones.

Se usan cuando se registran en el sistema grupos de estudiantes que ya cursaron o empezaron antes de que el sistema estuviera activo, o cuando hay un proceso de migración desde hojas de cálculo.

Nomenclatura de archivos

ParteEjemploDescripción
import_—Prefijo fijo que indica script de importación
XXXX0261Número de promoción (4 dígitos con cero a la izquierda)
_NXN1, N2Nivel del curso (N1 = Nivel 1, N2 = Nivel 2, etc.)
_parche_parche2Corrección posterior al script principal
_continuacion—Script que completa un anterior interrumpido

Ejemplos válidos

import_0261_N1.sql              → Promoción 0261, Nivel 1 (principal)
import_0262_N1.sql              → Promoción 0262, Nivel 1
import_0262_N1_parche.sql       → Corrección al script principal 0262-N1
import_0262_N1_parche2.sql      → Segunda corrección al script 0262-N1
import_0270_N1_continuacion.sql → Continuación de importación parcial 0270-N1

Estructura del script

Cada script sigue la misma estructura en bloques ordenados:

Encabezado del script

Comentario con metadatos: número de promoción, nivel, ciclo, fechas, estado y secuencias de referencia para las claves externas.

Configuración inicial

Selección de base de datos, charset y resolución del promo_id mediante SELECT (no se hardcodea el ID).

Referentes (opcional)

Si hay estudiantes que refirieron a otros en la promoción, se registran aquí sus bonificaciones/comisiones.

Bloque por estudiante

Cada estudiante tiene su propio bloque comentado. Contiene en orden: interesado → bitácora → estudiante → contacto → domicilio → salud → logística → educación → matrícula → formación → pagos → entregas.

Ejemplo anonimizado

A continuación se muestra la estructura completa con datos ficticios para ilustrar el patrón:

-- ============================================================
-- SCRIPT DE IMPORTACION: PROMOCION XXXX - N1
-- Hoja: XXXX-N1 | Promocion id=? | Ciclo EJEMPLO (id=?)
-- Fecha: YYYY-MM-DD al YYYY-MM-DD | Estado: finalizado
-- Secuencias desde: EFECXXX, TRANXXX, DEPOXXX, TARJX, BONOX
-- ============================================================

USE g_academico;
SET NAMES utf8mb4;

-- Resolucion dinamica del ID de promocion (no se hardcodea)
SET @promo_id = (
  SELECT id FROM promociones
  WHERE numero_promocion = 'XXXX' AND curso_id = 1
  LIMIT 1
);

-- ============================================================
-- REFERENTES (estudiantes que refirieron en esta promocion)
-- ============================================================
-- Ejemplo: cuando un estudiante activo refiere a un nuevo inscrito
-- se registra una bonificacion a su favor.
-- Si no hay referidos activos en la promo, se deja comentado.

-- SET @referente_id = (SELECT id FROM estudiantes WHERE documento_identidad = '0900000001' LIMIT 1);
-- INSERT INTO bonificaciones (estudiante_id, monto, tipo, estado, promocion_id, ...)
-- VALUES (@referente_id, 25.00, 'referido', 'activa', @promo_id, ...);

-- ============================================================
-- INTERESADOS, ESTUDIANTES, MATRICULAS, PAGOS Y ENTREGAS
-- ============================================================

-- ------ 1: APELLIDO NOMBRE EJEMPLO ------
-- Contexto: estudiante sin referido, descuento por evento especial
-- Precio lista $287.50, descuento 20% => total $230, saldo $0

-- 1.1 Registrar interesado (lead)
INSERT INTO interesados (
  nombres, apellidos, documento, telefono, email,
  fuente_origen_id, referido_por_id, estado_interes, created_at
) VALUES (
  'NOMBRE', 'APELLIDO EJEMPLO', '0900000001', '0999000001',
  'ejemplo.estudiante@correo.com', NULL, NULL, 'matriculado', 'YYYY-MM-DD 00:00:00'
);
SET @interesado_id = LAST_INSERT_ID();

-- 1.2 Registrar visita en bitácora
INSERT INTO bitacora_visitas (
  interesado_id, fecha_visita, observacion,
  convertido_a_prematricula, transferido
) VALUES (
  @interesado_id, 'YYYY-MM-DD 00:00:00', NULL, 1, 1
);
SET @bitacora_id = LAST_INSERT_ID();

-- 1.3 Asociar novedad a la visita (tipo de inscripción)
INSERT INTO bitacora_novedades (bitacora_id, novedad_id)
VALUES (
  @bitacora_id,
  (SELECT id FROM novedades
   WHERE categoria='nivel' AND subcategoria='Nivel 1'
   AND tipo_accion='inscripcion' LIMIT 1)
);

-- 1.4 Crear registro de estudiante
INSERT INTO estudiantes (
  centro_id, interesado_id, referente_id,
  nombres, apellidos, documento_identidad,
  nacionalidad, sexo, estado_academico
) VALUES (
  1, @interesado_id, NULL,
  'NOMBRE', 'APELLIDO EJEMPLO', '0900000001',
  'ECUATORIANA', 'F', 'matriculado'
);
SET @estudiante_id = LAST_INSERT_ID();

-- 1.5 Contacto del estudiante
INSERT INTO estudiantes_contacto (
  estudiante_id, email, telefono_celular,
  contacto_emergencia_nombre, contacto_emergencia_telefono
) VALUES (
  @estudiante_id, 'ejemplo.estudiante@correo.com',
  '0999000001', 'CONTACTO EMERGENCIA', '0999000002'
);

-- 1.6 Domicilio
INSERT INTO estudiantes_domicilio (
  estudiante_id, pais, provincia, canton, parroquia, direccion, codigo_postal

  @estudiante_id, 'ECUADOR', 'GUAYAS', 'GUAYAQUIL',
  'Guayaquil', 'DIRECCION EJEMPLO SN', '90001'

-- 1.7 Datos de salud

-- 1.8 Logística / disponibilidad

-- precio_lista $287.50 | desc_promo 20% ($57.50) | subtotal $230 | IVA $0 | total $230 | saldo $0

Truco: Secuencias en el encabezado

El encabezado del script declara desde qué número continúan las secuencias de referencia externas (ej: EFEC330). Esto permite rastrear el correlativo sin consultar la base de datos y evitar colisiones con scripts anteriores.

Casos especiales

Estudiante con referido

Si el estudiante viene referido por otro estudiante activo, se agrega el referente_id en el INSERT de estudiantes y se registra una bonificación al referente:

-- Obtener el ID del estudiante referente
SET @referente_id = (
  SELECT id FROM estudiantes
  WHERE documento_identidad = '0900000099' LIMIT 1
);

-- Registrar el estudiante con referente
INSERT INTO estudiantes (
  centro_id, interesado_id, referente_id, ...
) VALUES (1, @interesado_id, @referente_id, ...);

Matrícula con descuento individual

Cuando hay un descuento adicional al descuento de promoción (negociado individualmente):

-- precio_lista $287.50 | desc_promo 10% | desc_individual $20
-- subtotal = 287.50 - 28.75 - 20 = 238.75
INSERT INTO matriculas (
  ...,
  precio_lista,         -- 287.50
  desc_promocion_porcentaje, -- 10.00
  desc_promocion_monto, -- 28.75
  descuento_individual, -- 0.0  (porcentaje, si aplica)
  descuento_individual_monto, -- 20.00
  subtotal,             -- 238.75
  impuestos,            -- 0.0
  total_a_pagar,        -- 238.75
  saldo,                -- saldo pendiente
  ...
) VALUES (...);

Matrícula con bonificación

-- La bonificacion se registra en tabla bonificaciones y afecta el saldo
INSERT INTO bonificaciones (
  matricula_id, monto, tipo, estado, fecha_inicio, fecha_fin, descripcion
) VALUES (
  @matricula_id, 30.00, 'descuento_especial', 'aplicada',
  'YYYY-MM-DD', 'YYYY-MM-DD', 'Bono lanzamiento'
);

Estudiante sin entrega de kit

Si el kit no fue entregado en la misma fecha (ej: talla pendiente), el estado va como 'pendiente' y la fecha_entrega queda en NULL:

INSERT INTO entregas (
  matricula_id, entregable_id, cantidad, talla, estado, fecha_entrega
) VALUES (
  @matricula_id,
  (SELECT id FROM catalogo_entregables WHERE nombre='Camisa' LIMIT 1),
  1, NULL, 'pendiente', NULL
);

Script de parche

Un _parche.sql corrige errores específicos del script principal. Siempre debe identificar el registro por clave única (no por secuencial):

-- PARCHE: corregir talla de camisa para documento 0900000001
UPDATE entregas e
JOIN matriculas m ON e.matricula_id = m.id
JOIN estudiantes est ON m.estudiante_id = est.id
SET e.talla = 'L'
WHERE est.documento_identidad = '0900000001'
  AND e.entregable_id = (
    SELECT id FROM catalogo_entregables WHERE nombre='Camisa' LIMIT 1
  );

Consultas de verificación previa

Antes de construir o ejecutar un script de importación, se recomienda correr estas consultas de diagnóstico para evitar duplicados, colisiones de secuencias y errores de integridad.

Solo para uso interno

Estas consultas contienen lógica sobre la estructura interna de la base de datos. Reemplaza {NUMERO} con el número de promoción real y los documentos del bloque IN (...) con los de la hoja de trabajo correspondiente. No publicar con datos reales de estudiantes.

A. Estado actual de la promoción

Verifica que la promoción existe y su estado antes de importar.

-- A. Estado actual de la promoción (si ya existe en BD)
SELECT id, numero_promocion, codigo_convocatoria,
       fecha_inicio, fecha_fin, estado
FROM promociones
WHERE numero_promocion = '{NUMERO}';

B. Últimas secuencias de referencia externa

Obtiene los últimos valores de referencia por método de pago para continuar la numeración sin colisiones.

-- B. Últimas referencias externas por tipo de pago
-- Resultado esperado (ejemplo):
-- EFEC0000527 | TRAN0000308 | TARJ0000011 | DEPO0000074 | BONO0000004

SELECT
  MAX(CASE WHEN referencia_externa LIKE 'EFEC%' THEN referencia_externa END) AS ultimo_efec,
  MAX(CASE WHEN referencia_externa LIKE 'TRAN%' THEN referencia_externa END) AS ultimo_tran,
  MAX(CASE WHEN referencia_externa LIKE 'TARJ%' THEN referencia_externa END) AS ultimo_tarj,
  MAX(CASE WHEN referencia_externa LIKE 'DEPO%' THEN referencia_externa END) AS ultimo_depo,
  MAX(CASE WHEN referencia_externa LIKE 'BONO%' THEN referencia_externa END) AS ultimo_bono
FROM pagos;
¿Por qué importa este paso?

El encabezado de cada script declara desde qué número continúan las secuencias (ej: EFEC330). Si otro script se ejecutó antes, los números habrán avanzado. Esta consulta confirma el valor real en BD antes de escribir el script.

C. Estudiantes ya ingresados en la promoción

Lista los estudiantes que ya tienen matrícula en esa promoción (para no duplicarlos).

-- C. Estudiantes ya registrados en la promoción {NUMERO}
SELECT e.documento_identidad,
       CONCAT(e.apellidos, ' ', e.nombres) AS nombre
FROM matriculas m
JOIN estudiantes e ON e.id = m.estudiante_id
WHERE m.promocion_id = (
  SELECT id FROM promociones
  WHERE numero_promocion = '{NUMERO}' LIMIT 1
);

D. Verificar existencia previa de estudiantes de la hoja

Antes de insertar, identifica cuáles documentos de la hoja de trabajo ya existen como estudiantes en el sistema (pueden ser reprobados que reingresan o estudiantes de otras promociones).

-- D. Verificar cuáles documentos ya existen como estudiantes
-- Reemplazar la lista IN (...) con los documentos de la hoja de trabajo
SELECT documento_identidad, id,
       CONCAT(apellidos, ' ', nombres) AS nombre
FROM estudiantes
WHERE documento_identidad IN (
  '0900000001', '0900000002', '0900000003', '0900000004',
  '0900000005', '0900000006', '0900000007', '0900000008',
  '0900000009', '0900000010', '0900000011', '0900000012'
  -- ... agregar todos los documentos de la hoja
);

E. Verificar matrículas previas de los estudiantes existentes

Para los que ya existen, verifica si tienen matrícula previa (reprobaron un nivel anterior, se cambiaron de promoción, etc.).

-- E. Matrículas previas de estudiantes que ya existen en BD
SELECT e.documento_identidad,
       CONCAT(e.apellidos, ' ', e.nombres) AS nombre,
       m.codigo_matricula, m.estado_matricula,
       p.numero_promocion
FROM matriculas m
JOIN estudiantes e  ON e.id  = m.estudiante_id
JOIN promociones p  ON p.id  = m.promocion_id
WHERE e.documento_identidad IN (
  '0900000001', '0900000002', '0900000003', '0900000004',
  '0900000005', '0900000006', '0900000007', '0900000008'
  -- ... agregar los documentos que el paso D indicó como existentes
)
ORDER BY e.documento_identidad, m.fecha_matricula;
Interpretación del resultado de E

Si un estudiante aparece con estado_matricula = 'activa' en otra promoción del mismo nivel, es un caso especial (cambio de grupo o reingreso). En ese caso no se usa el bloque completo de INSERT de estudiante — solo se insertan la matrícula nueva y sus pagos referenciando el @estudiante_id ya existente.

Checklist antes de ejecutar

Buena práctica

Siempre ejecuta el script en un entorno de prueba antes de correrlo en producción. Verifica que la promoción existe y que las secuencias no colisionan con registros anteriores.

  • Correr consultas de verificación A–E y revisar resultados
  • Verificar que la promoción (numero_promocion) existe en la tabla promociones
  • Confirmar que el curso_id es correcto para el nivel a importar
  • Revisar que las secuencias del encabezado continúan desde el último valor en BD (resultado B)
  • Identificar estudiantes que ya existen (resultado D) y ajustar sus bloques en el script
  • Ejecutar en transacción: START TRANSACTION; y verificar antes de COMMIT;
  • Corroborar conteo de estudiantes insertados vs. total esperado
  • Validar que los saldos finales de matrículas son correctos
