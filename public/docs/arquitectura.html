Arquitectura del sistema

Componentes, flujo de datos, autenticación, patrones de código y decisiones técnicas de Ademy.

Resumen

Ademy es una aplicación web full-stack en arquitectura cliente-servidor. El frontend React se comunica con el backend Express mediante una API REST protegida por JWT. El backend persiste datos en MySQL y almacena archivos en el filesystem local o en Cloudflare R2.

Flujo de un request

Frontend Component
  └─► Service (httpClient.js)         ← agrega Bearer token automáticamente
        └─► Backend Route              ← Express router
              └─► auth.middleware      ← verifyToken + checkPermission
                    └─► Controller     ← lógica de negocio, validaciones
                          └─► Model    ← queries MySQL (mysql2 pool)
                                └─► MySQL

Autenticación y autorización

Login

El usuario envía credenciales a POST /auth/login. El backend valida, genera un JWT con 4h de expiración y almacena el hash del token en la tabla sesiones.

Requests protegidos

El frontend inyecta el header Authorization: Bearer <token> en cada request. verifyToken valida la firma JWT y verifica que el hash esté en sesiones (para soportar logout).

Verificación de permisos (RBAC)

checkPermission('codigo') carga los permisos del rol del usuario y verifica que tenga el permiso requerido. El usuario con el rol de mayor jerarquía bypasea todos los checks de permisos.

Logout

POST /auth/logout elimina el registro de sesiones, invalidando el token aunque no haya expirado. El frontend borra el token de localStorage ante cualquier 401.
